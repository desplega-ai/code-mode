# code-mode — Brainstorm

## Context

**What**: An MCP server (initially for Claude Code) that gives agents SDK awareness and reusable script management. The core idea is that agents should be able to discover what SDKs/APIs are available, write reusable TypeScript scripts against them, and build a persistent knowledge base of scripts that survives across sessions.

**MVP scope** (as stated by Taras):
1. **SDK discovery** — Auto-generate SDK definitions from installed MCPs for Claude Code, plus user-defined SDKs in `~/.code-mode/sdks/<name>/` and `.code-mode/sdks/<name>/`
2. **Script CRUD + execution** — Create/run/query TS scripts in `<code-mode>/scripts/<name>.ts`, executed via Bun (default) or Node+ts
3. **Type querying** — Search and browse types, interfaces, functions, classes from discovered SDKs
4. **Cross-session persistence** — Scripts persist and form a reusable knowledge base

**Future ideas** (noted, not in MVP):
- Sandboxing (just-bash, e2b, modal)
- Skills for harnesses
- Hook system for Claude integration
- Claude marketplace plugin

**Prior research**: `thoughts/taras/research/2026-04-12-ts-code-analysis-options.md` — comprehensive analysis of ts-morph, oxc-parser, tree-sitter, LSP, and Bun compatibility for programmatic TS analysis. Key finding: ts-morph is the recommended primary tool for type resolution.

## Exploration

### Q: What's the core pain you're solving?
**Answer**: Primarily reusability — a way to have compounding learnings and context usage (tokens and cost) reduction. Agents keep writing the same scripts from scratch each session. The goal is that work compounds over time rather than being thrown away.

**Insights**: This frames code-mode as an **agent efficiency tool**, not just a developer tool. The value proposition is economic: reduce token spend by reusing proven scripts instead of regenerating them. SDK awareness is a means to that end — you need to know what's available to write good reusable scripts against it. The "compounding learnings" angle suggests this is about building institutional knowledge that any agent session can tap into.

### Q: Who authors and who consumes scripts?
**Answer**: Human + agent both author. Any agent can consume. Taras might write a script manually, or an agent might create one during a session. Either way, any future session/agent should be able to discover and use it.

**Insights**: This means the script format must be human-writable AND agent-writable. It needs to be discoverable without reading every file — some kind of index/metadata. The "any agent consumes" requirement means scripts can't depend on agent-specific context; they need to be self-contained with clear inputs/outputs and documentation.

### Q: What is a "script" in code-mode?
**Answer**: It's pretty much what the agent thinks it should be. Generally something that runs and outputs something, but could also run in the background. Motivation comes from Cloudflare's code-mode blog post.

**Cloudflare code-mode reference**: Cloudflare's approach converts MCP tools into TypeScript APIs and presents the agent with a single "execute code" tool. The agent writes TS code that calls those APIs, runs in a sandbox, and returns results via `console.log()`. Key insight from their post: "LLMs are better at writing code to call MCP, than at calling MCP directly" — because LLMs have massive TS training data but limited tool-call examples. Also: chaining multiple calls as code avoids round-tripping each intermediate result through the LLM.

**Insights**: Taras's vision extends Cloudflare's concept in a critical way: **persistence and reuse**. Cloudflare's code-mode generates throwaway code per-request. Taras wants the generated code to **accumulate** as a reusable library. This is the key differentiator. The scripts are both the mechanism (how work gets done) and the artifact (what compounds over time).

### Q: How does the agent discover existing scripts?
**Answer**: Scripts should NOT become MCP tools — that defeats the purpose of reducing MCP bloat. Instead, discovery should be a mix of search types (name, description, type signatures, semantic) that returns relevant chunks/files depending on the query.

**Insights**: This is a strong design constraint. The MCP server exposes a small, fixed set of tools (search, run, create, type-query) rather than dynamically registering each script as a tool. The search API becomes the primary interface — it needs to be good enough that an agent can find the right script without browsing. This aligns with the research findings: ts-morph can extract type signatures to enable type-aware search ("find a function that takes `string` and returns `Promise<GithubPR[]>`"). The search layer is essentially a code-aware knowledge base.

### Q: MCP auto SDK — what exactly gets generated?
**Answer**: The auto SDK should generate types AND callable functions, not just type definitions. Types should be usable outside of scripts too (e.g. in the agent's own reasoning about data shapes).

**Insights**: This means the SDK generator produces a full module: interfaces for inputs/outputs + wrapper functions that actually call the MCP tools. The generated SDK becomes a first-class TypeScript library, not just `.d.ts` stubs. This is exactly what Cloudflare does — but here the generated SDK persists on disk and can be imported by any script.

### Q: What's the execution model constraint?
**Answer**: **Only type-safe code can be run.** This is a core design principle — like giving agents functional programming. Example: the GitHub MCP returns huge JSON lists. Instead of dumping that into the LLM context, you write a type-safe script that filters/transforms the data, maybe composing another script that does fuzzy matching.

**Insights**: This is the most important design decision so far. Type safety as a hard constraint means:
1. **Scripts must typecheck before execution** — ts-morph or `tsc --noEmit` as a gate
2. **Composition is first-class** — scripts import and call other scripts, building up a functional pipeline
3. **The token savings are real** — instead of the LLM processing a 500-item JSON response, a script filters it to 5 items, and only those 5 enter the context
4. **Error messages become useful** — type errors tell the agent exactly what's wrong, unlike runtime crashes in untyped code
5. **The SDK types serve double duty** — they're both the contract for execution AND the documentation for discovery (agents can search by type signature)

This reframes the entire system: code-mode isn't just "save and run scripts" — it's a **typed computation layer** that sits between MCP tools and the LLM, reducing what flows through the context window.

### Q: What's the execution flow?
**Answer**: Both write-new AND call-existing, plus composition. Agents can write new TS code that imports existing scripts, typecheck, and run. All three patterns are first-class.

**Insights**: Three execution modes for the MCP `run` tool:
1. **Inline**: Agent writes fresh TS code → typecheck → execute → return output
2. **Call**: Agent references existing script by name + passes args → execute → return output  
3. **Compose**: Agent writes new code that `import`s existing scripts → typecheck → execute → return output

Mode 3 is the most powerful — it's how the knowledge base compounds. An agent writes a "filter GitHub PRs by author" script once. Later, another agent writes a new script that imports that filter + a fuzzy matcher + a formatter, creating a higher-level capability. The type system ensures these compositions are safe.

### Q: What MCP tools does code-mode expose?
**Answer**: Minimal — lean on the agent's existing Read tool for deep dives. Code-mode provides pointers (search results, file paths, type summaries), and the agent uses Read to inspect source when needed. ~4-5 tools total.

**Insights**: This is a smart design. The MCP tools are a thin orchestration layer:
- `search` — find scripts/types by name, description, type signature, semantic query. Returns pointers (paths, summaries), not full source.
- `run` — execute inline code or a named script. Typechecks first. Returns output.
- `save` — persist a script with metadata (name, description, tags, input/output types).
- `list_sdks` — show available SDKs (auto-generated from MCPs + user-defined).
- `query_types` — search SDK types/interfaces/functions. Returns type signatures.

The agent already has Read, so code-mode doesn't need `get_script_source` — just return the file path and let the agent read it. This keeps the tool surface minimal and avoids duplicating capabilities. It also means code-mode works well with any agent that has file access, not just Claude Code.

### Q: What metadata does a saved script carry?
**Answer**: Minimal + usage tracking. Name, description, tags, input/output types, plus run count, last run date, success/failure rate. Quality tiers (draft/tested/promoted) are overkill for MVP.

**Insights**: Usage tracking is cheap to implement and high value for search ranking — a script that's been run 50 times successfully is a better search result than one that's been run once and failed. The metadata could live as YAML frontmatter in the script file itself (human-readable, git-friendly) or in a separate index. Frontmatter is cleaner — the script IS its own documentation:

```typescript
/**
 * @name filter-github-prs
 * @description Filter GitHub PRs by author and status
 * @tags github, pr, filter
 * @runs 47
 * @lastRun 2026-04-12
 * @successRate 0.96
 */
export function filterPRs(prs: GithubPR[], author: string, status?: string): GithubPR[] { ... }
```

Actually — JSDoc + a sidecar `.meta.json` might be cleaner. The TS file stays pure TS (no magic comments), and the metadata is machine-managed. The agent sees both via search results.

### Q: Should scripts import across scopes (global vs project-local)?
**Answer**: Either merged namespace or global→local imports. Unsure which is better.

**Claude's take**: I'd recommend **option 1 (global→local, not reverse)** over merged namespace. Here's why:

- **Merged namespace** creates ambiguity: if both global and project have `filter-prs.ts`, which one wins? You'd need conflict resolution rules.
- **Global→local** is clean: global scripts are "stdlib" — shared utilities. Project scripts are project-specific and can build on the stdlib. No ambiguity.
- **Import syntax is natural**: `import { filterPRs } from "@code-mode/global/filter-prs"` vs `import { parseConfig } from "./parse-config"`. The `@code-mode/global/` prefix makes scope explicit.
- **Search merges both**: when the agent searches, it sees results from both scopes with a scope indicator. No confusion.

This mirrors how node_modules work: global packages are available everywhere, local packages are project-scoped. Familiar pattern.

### Q: How do scripts produce output?
**Answer**: Both. Typed return value is primary, console.log is captured as side-channel.

**Insights**: This gives two output channels:
1. **Primary (structured)**: Script exports a main function with a typed return value. Code-mode serializes it to JSON and returns it to the agent. The agent can reason about the structure because it knows the return type from the type system.
2. **Side-channel (logs)**: `console.log()` output is captured and returned alongside the structured output. Useful for debugging, progress messages, or when the script wants to output human-readable text.

The MCP response from `run` could look like:
```json
{
  "result": { ... },      // serialized return value
  "resultType": "GithubPR[]",  // type annotation
  "logs": ["Fetched 47 PRs", "Filtered to 5"],  // console output
  "success": true,
  "durationMs": 342
}
```

This is strictly better than Cloudflare's console.log-only model — agents get structured data they can pipe into the next operation without re-parsing.

## Synthesis

### Key Decisions

1. **Core problem**: Compounding learnings + token/cost reduction. Scripts accumulate as reusable knowledge, not throwaway code.
2. **Authorship**: Both humans and agents author scripts. Any agent session can discover and use them.
3. **Type safety is mandatory**: Only type-checked code can execute. This is the core design constraint — agents get functional-style programming with type guarantees.
4. **Composition is first-class**: Three execution modes — write inline, call existing, compose (new code importing existing scripts).
5. **Minimal MCP surface**: ~5 tools (search, run, save, list_sdks, query_types). Lean on agent's existing Read tool for deep dives. Scripts do NOT become MCP tools.
6. **SDK generation**: Auto-generate typed SDKs (types + callable functions) from MCP server schemas. Types are usable outside scripts too.
7. **Project-only scope (MVP)**: Single `.code-mode/` per project. Global `~/.code-mode/` is v2 (avoids tsconfig complexity + multi-team conflicts).
8. **Dual output**: Typed return values (primary, structured) + console.log (side-channel, captured).
9. **Metadata**: JSDoc in TS files + usage tracking (run count, last run, success rate) in SQLite index.
10. **CLI-first**: Ship as `code-mode` CLI with subcommands (`mcp`, `reindex`, `doctor`, `run`, etc.). MCP server is one mode of the CLI, not the only interface.
11. **Package workspace**: Each scope (global/project) has a `package.json` enabling `bun install` for third-party deps and `@/...` import aliases.
12. **Type safety hooks**: On script edit, re-typecheck script + dependents. Broken scripts marked `unusable` in index, deprioritized in search.
13. **Stdlib seeding**: Ship with utility stdlib + auto-generate MCP starter templates on `code-mode init`.

### Open Questions

- **Metadata format**: ✅ RESOLVED — JSDoc in the TS files. Keep it simple.
- **Distribution model**: ✅ RESOLVED — Ship as CLI first (`code-mode`) with subcommands: `mcp` (start MCP server), `reindex` (rebuild search index), `doctor` (check health), etc. CLI-first enables standalone use and debugging.
- **Search implementation**: ✅ RESOLVED — SQLite + FTS5 for script index. ts-morph extracts type signatures at index time.
- **SDK generation from MCPs**: ✅ RESOLVED — Check Claude Code's MCP docs at code.claude.com/docs/en/mcp.md for config format. Need to research how to read MCP config and call `tools/list`.
- **Typecheck performance**: ✅ RESOLVED — Incremental checking preferred. Not thousands of files (scripts are small library, not a codebase). Could explore oxc-parser for fast structural validation + ts-morph for targeted type resolution (hybrid from research).
- **Import resolution**: ✅ RESOLVED — `@code-mode/global/...` path alias pattern. Generated `tsconfig.json` with path mappings.
- **Versioning**: ✅ RESOLVED — Global dir (`~/.code-mode/`) is an auto-managed git repo with auto-commits on save/edit. Claude hooks can trigger version snapshots on edit. Project-level `.code-mode/` relies on the project's own git.
- **Error recovery**: ✅ RESOLVED — Return type error messages directly. No suggestions needed for MVP.
- **Sandboxing**: ✅ RESOLVED — MVP runs locally via Bun. Sandboxing (e2b, modal, just-bash) is future work only.

### Constraints Identified

- **Bun as primary runtime**: Fast startup (8-15ms), native TS execution, Bun.Transpiler.scan() for fast import extraction.
- **ts-morph for type analysis**: Full type resolution, works in Bun, required for type-safe execution gate and type-aware search.
- **MCP protocol limits**: Small fixed tool surface. No dynamic tool registration per script.
- **File system is the source of truth**: Scripts are files on disk, git-friendly, human-editable. No database for MVP.
- **Agent context window**: Output must be concise. Structured JSON returns are preferable to dumping raw data.

### Core Requirements (MVP)

**P0 — Must have:**
1. **`run` tool**: Execute inline TS code or a named script. Typecheck before execution. Return typed result + logs.
2. **`save` tool**: Persist a script with metadata. Validate it typechecks before saving.
3. **`search` tool**: Find scripts by name, description, tags, or type signatures. Return pointers (paths, summaries).
4. **`list_sdks` tool**: Show available SDKs with their exported types/functions.
5. **`query_types` tool**: Search SDK types, interfaces, functions by name or signature pattern.
6. **SDK auto-generation**: Read MCP schemas → generate typed TS SDK modules with callable functions.
7. **User-defined SDKs**: Load from `~/.code-mode/sdks/` and `.code-mode/sdks/`.
8. **Global + project scoping**: Two scopes, global→local imports.
9. **Type safety gate**: Scripts must pass type checking to execute or save.

**P1 — Should have:**
- Usage tracking (run count, success rate)
- Script composition (import other scripts)
- Console.log capture alongside typed returns

**P2 — Nice to have (not MVP):**
- Sandboxed execution (e2b, modal, just-bash)
- Semantic search over scripts
- Claude marketplace plugin
- Hook system for Claude auto-discovery
- Skills for harnesses

**Clarifications from review:**
- `sdks/` = human/agent-authored SDK modules (custom wrappers, utilities, API clients)
- `sdks/.generated/` = auto-generated from MCP schemas (read-only, regenerated via `code-mode reindex`)
- `package.json` at root of each scope = Bun workspace for `bun install`, enables `@/...` style imports and third-party deps
- Scripts import via: `import { ... } from “@/sdks/my-utils”` or `import { ... } from “@code-mode/global/scripts/fuzzy”`

### Tech Stack

| Component | Technology | Rationale |
|-----------|-----------|-----------|
| Runtime | Bun | Fast startup, native TS, Bun.Transpiler for imports |
| MCP Server | `@modelcontextprotocol/sdk` | Standard MCP server library |
| Type analysis | ts-morph | Full type resolution, good DX, Bun compatible |
| Search index | SQLite + FTS5 | Lightweight, no server, fast text search |
| Script execution | Bun subprocess | `Bun.spawn(["bun", "run", script])` with typed wrapper |
| SDK generation | ts-morph + custom codegen | Generate TS from MCP tool schemas |

### Post-Synthesis Review (Taras + Claude)

**1. Cold start — stdlib seeding proposals:**

Option A: **Utility stdlib** — Ship with 10-15 generic scripts:
- `transform/filter.ts` — type-safe array filtering with predicate
- `transform/map-pick.ts` — pick specific fields from objects
- `transform/group-by.ts` — group arrays by key
- `transform/sort-by.ts` — sort with type-safe comparator
- `text/fuzzy-match.ts` — fuzzy string matching (Levenshtein or similar)
- `text/truncate.ts` — smart truncation with ellipsis
- `json/flatten.ts` — flatten nested JSON to dot-notation paths
- `json/diff.ts` — diff two JSON objects, return changes
- `format/table.ts` — format array of objects as markdown table
- `format/summary.ts` — summarize large arrays (count, sample, types)

Option B: **MCP-aware templates** — Auto-generate starter scripts per connected MCP:
- When GitHub MCP is connected → scaffold `github/list-my-prs.ts`, `github/pr-summary.ts`
- When Slack MCP → scaffold `slack/search-messages.ts`, `slack/channel-summary.ts`
- Templates are thin wrappers that demonstrate the SDK usage pattern

Option C: **Both** — Ship utility stdlib + generate MCP templates on `code-mode init`

**Recommendation**: Option C. The stdlib provides immediate value; the MCP templates teach agents how to use the generated SDKs.

**2. Script quality / gc:**
- `code-mode doctor` includes stale script detection (unused for N days, low success rate)
- `code-mode gc` suggests scripts to remove or consolidate (near-duplicates by type signature)
- Part of the regular health check flow

**3. MCP SDK weak types:**
- Acknowledged — not our problem. If an MCP returns `any`, the generated SDK will have `any`. Users can enhance types in `sdks/` (human-authored) to wrap the generated SDK with better types.

**4. Scope simplification for MVP:**
- **Start project-only (`.code-mode/`)** — skip global `~/.code-mode/` in MVP
- Avoids tsconfig complexity, multi-person team conflicts, and global state management
- Global scope is a clear v2 feature once the project-local model is proven
- Updated architecture: single `.code-mode/` directory per project

**5. Type safety enforcement via hooks:**
- On script edit (via Claude hooks or file watcher): re-typecheck the modified script AND its dependents
- If typecheck fails: mark script as `unusable` in metadata/index (SQLite `status` field)
- `code-mode doctor` reports broken scripts
- `search` tool deprioritizes or excludes `unusable` scripts from results
- Agent gets clear signal: "this script exists but is currently broken"
