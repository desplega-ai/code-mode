# code-mode — Implementation Plan

## Overview

Ship `code-mode` as a single Bun CLI that also runs as an MCP server (`code-mode mcp`). Agents discover SDKs, write TypeScript that typechecks before execution, and build up a reusable script knowledge base under `.code-mode/` in each project. Five MCP tools — `search`, `run`, `save`, `list_sdks`, `query_types` — expose the functionality; a SQLite+FTS5 index powers discovery; ts-morph enforces the type-safety gate.

## Current State

Greenfield repo. Only `thoughts/`, `LICENSE`, and a stub `README.md` exist. No `package.json`, no source, no tests. All design decisions are captured in the brainstorm (`thoughts/taras/brainstorms/2026-04-12-code-mode.md`) and research doc (`thoughts/taras/research/2026-04-12-ts-code-analysis-options.md`); this plan turns them into code.

## Desired End State

A user can:

1. `bun install -g code-mode` (or `bunx code-mode`) and run `code-mode init` inside any project.
2. Register `code-mode mcp` with Claude Code as an MCP server.
3. Have an agent discover generated SDKs for their installed MCPs, write a type-safe script, save it, and have it surface in `search` on the next session.
4. Run `code-mode doctor` to see health; `code-mode gc` to identify stale/duplicate scripts.
5. Run `code-mode inspect` to open a browser-based debugging UI for every configured MCP (inspired by `@mcp-use/inspector`) — list tools/resources/prompts, invoke tools live, view generated SDKs.

All surface area is exercisable from the CLI standalone — the MCP server is a thin adapter, not a separate implementation. The inspector ships as a separate workspace package, lazy-loaded, so it doesn't bloat the core install.

## Non-Goals (MVP)

- Global scope (`~/.code-mode/`) — project-only for MVP, per brainstorm Section 4 "Post-Synthesis Review".
- Sandboxed execution (e2b, modal, just-bash).
- Semantic/vector search — FTS5 only.
- Auto-healing of broken scripts. `doctor` reports them; fixing is on the agent/human.
- Fixing weak MCP schemas. If upstream says `any`, we emit `unknown` and move on.

## Tech Stack

| Concern | Choice | Rationale |
|---|---|---|
| Runtime | Bun | Fast startup (8–15ms), native TS, required by brainstorm |
| Type analysis | `ts-morph` (primary) | Full type resolution, Bun-compatible (pure JS, confirmed by devs). Per research Tier 1 recommendation. |
| Type analysis fallback | `@ts-morph/bootstrap` (48kB) | Available if wrapper overhead becomes an issue on large indexes |
| Storage | `bun:sqlite` + FTS5 | Native to Bun, no npm dep, FTS5 shipped. Schema evolved via a tiny hand-rolled migration runner (versioned `.sql` files + `schema_version` table). Drizzle considered and deferred — see open tracks. |
| MCP server | `@modelcontextprotocol/sdk` | Standard library |
| CLI parsing | `commander` | Battle-tested, small enough for MVP |
| Tests | `bun:test` | Native to Bun, no extra dep |

### Performance knobs (from research)

- `useInMemoryFileSystem: true` for unit tests (~30ms vs ~80ms).
- Avoid `getExportedDeclarations()` in hot paths (~1s overhead, issue #644) — use `getDescendantsOfKind(ExportKeyword)` + lazy resolve.
- `forgetNodesCreatedInBlock()` after each file in the bulk indexer to bound memory.
- Import `typescript` as `import * as ts from "typescript"` (not named imports) to avoid CJS interop.

### Testing Strategy

- **Runner**: `bun test` (native). One `*.test.ts` per module; integration suites live in `test/e2e/`.
- **Fixtures**: every suite operating on a workspace points at `test/fixtures/*` copied into `process.cwd()/.code-mode-test-<pid>/` (torn down in `afterAll`). No shared global state.
- **In-memory FS**: `test/analysis/` uses ts-morph's `useInMemoryFileSystem: true` (~30ms startup) — real-FS only where we need to spawn subprocesses.
- **Golden tests**: generated SDK TS files in `test/fixtures/snapshots/` are compared byte-for-byte; schema-driven codegen is the most regression-prone area.
- **Coverage target (not a gate)**: 80% for `src/analysis`, `src/runner`, `src/sdk-gen`; lower expectations for CLI wiring.

## Phase 2 — `code-mode init` + workspace scaffolding + stdlib seed

### Goal
`code-mode init` scaffolds a complete, typecheck-clean `.code-mode/` workspace. Seeds 4 stdlib utilities to prove the mechanism end-to-end.

### Changes

1. **`src/commands/init.ts`** — implements:
   - Check target dir (`process.cwd()` by default, or `--path`).
   - If `.code-mode/` exists and `--force` not passed, bail with a clear message.
   - Write layout: `package.json`, `tsconfig.json`, `scripts/.keep`, `sdks/.keep`, `sdks/stdlib/*.ts`, `sdks/.generated/.keep`, empty `code-mode.db` (via `new Database(path)`).
   - Run `bun install` inside `.code-mode/` (so `@/*` paths resolve; stdlib may later import `typescript` types etc.). Skip if `--no-install`.
2. **`src/templates/`** — templates emitted as string literals from TS modules. Avoids any packaging gymnastics for MVP.
   - `tsconfig.json.ts` emits:
     ```json
     {
       "compilerOptions": {
         "target": "esnext",
         "module": "preserve",
         "moduleResolution": "bundler",
         "strict": true,
         "noEmit": true,
         "types": ["bun-types"],
         "baseUrl": ".",
         "paths": { "@/*": ["./*"] }
       },
       "include": ["scripts/**/*.ts", "sdks/**/*.ts"]
     }
     ```
   - `package.json.ts` emits a minimal workspace root.
3. **Stdlib seeds (`src/templates/stdlib/*`)** — 4 files, each with JSDoc `@name`, `@description`, `@tags`, typed signatures:
   - `filter.ts` — `filter<T>(items: T[], predicate: (item: T) => boolean): T[]`
   - `fuzzy-match.ts` — `fuzzyMatch(query: string, candidates: string[], limit?: number): string[]`
   - `flatten.ts` — `flatten(obj: unknown, prefix?: string): Record<string, unknown>`
   - `table.ts` — `table<T extends Record<string, unknown>>(rows: T[], columns?: (keyof T)[]): string`

## Phase 3 — Type analysis module (ts-morph wrapper)

### Goal
A reusable in-process analysis module. Typecheck any workspace file; extract structured info (exports, signatures, JSDoc) for the indexer.

### Changes

1. **`src/analysis/project.ts`** — `loadProject(workspaceDir: string): Project`. Loads `.code-mode/tsconfig.json`. Exposes a `scopedExtract` helper that calls `forgetNodesCreatedInBlock()` internally so callers can't forget.
2. **`src/analysis/typecheck.ts`** — `typecheckFile(project, filePath): Diagnostic[]` and `typecheckAll(project): Map<string, Diagnostic[]>`. Diagnostic shape: `{ file, line, col, code, message, severity }` — no raw ts-morph types escape this module.
3. **`src/analysis/extract.ts`** — `extractExports(project, filePath): ExportInfo[]`. `ExportInfo = { name, kind: 'function'|'type'|'interface'|'class'|'const', signature: string, jsdocDescription?: string, jsdocTags?: {name, value}[] }`. Uses `getDescendantsOfKind(ExportKeyword)` + resolves types lazily per research guidance.
4. **Tests** (`test/analysis/*.test.ts`) using `useInMemoryFileSystem: true`:
   - Valid file extracts all export kinds.
   - Broken file returns diagnostics with expected shape.
   - JSDoc description + custom tags flow through.
   - Memory smoke test: extract over 50 synthetic files; heap growth bounded (spot check via `process.memoryUsage()`).

### Success Criteria:

## Phase 4 — Storage + FTS5 index + `reindex` + `list_sdks` + `query_types`

### Goal
SQLite-backed index over scripts and SDK symbols, keyed by filesystem state. `reindex` rebuilds it; `list-sdks` and `query-types` CLI commands query it (wiring the same functions that the MCP tools will use in Phase 7).

### Changes

1. **`src/db/migrations/`** — versioned `.sql` files (`001_initial.sql`, `002_<next>.sql`, …). A 40-line migration runner in `src/db/migrate.ts`: reads `PRAGMA user_version`, applies any pending files in a transaction, bumps the pragma. No ORM. Swap-in Drizzle remains a post-MVP option (see open tracks).
2. **`src/db/schema.ts`** — initial schema lives in `src/db/migrations/001_initial.sql`:
   ```sql
   CREATE TABLE IF NOT EXISTS scripts (
     path TEXT PRIMARY KEY,
     name TEXT NOT NULL,
     description TEXT,
     tags TEXT,                      -- JSON array
     exports_json TEXT,              -- serialized ExportInfo[]
     status TEXT NOT NULL DEFAULT 'ok',  -- 'ok' | 'unusable'
     status_reason TEXT,
     runs INTEGER NOT NULL DEFAULT 0,
     last_run TEXT,
     success_rate REAL,
     indexed_at TEXT NOT NULL
   );

   CREATE VIRTUAL TABLE IF NOT EXISTS scripts_fts USING fts5(
     name, description, tags, signatures, content=scripts, content_rowid=rowid
   );

   CREATE TABLE IF NOT EXISTS symbols (
     id INTEGER PRIMARY KEY AUTOINCREMENT,
     source_path TEXT NOT NULL,
     kind TEXT NOT NULL,
     name TEXT NOT NULL,
     signature TEXT NOT NULL,
     jsdoc TEXT,
     scope TEXT NOT NULL,             -- 'stdlib' | 'sdk' | 'generated' | 'script'
     sdk_name TEXT
   );

   CREATE VIRTUAL TABLE IF NOT EXISTS symbols_fts USING fts5(
     name, signature, jsdoc, content=symbols, content_rowid=id
   );

   CREATE TABLE IF NOT EXISTS sdks (
     name TEXT PRIMARY KEY,
     scope TEXT NOT NULL,             -- 'stdlib' | 'user' | 'generated'
     source_dir TEXT NOT NULL,
     symbol_count INTEGER NOT NULL DEFAULT 0,
     last_indexed TEXT NOT NULL
   );
   ```
3. **`src/db/repo.ts`** — prepared-statement helpers for upsert / delete / query. All mutations in a transaction per reindex run.
4. **`src/index/reindex.ts`** — `reindex(workspaceDir, opts): ReindexReport`:
   - Load project (Phase 3).
   - Walk `sdks/` (recursively), `scripts/` (flat for now); for each `.ts`:
     - Run `extractExports`.
     - Upsert `scripts` or `symbols` row based on location.
   - Delete rows whose `path` no longer exists.
   - FTS5 triggers keep the virtual tables in sync via `content=` tables — populate after the base tables are updated.
5. **`src/commands/reindex.ts`** — CLI entry. Supports `--paths <a,b,c>` for targeted reindex (used later by edit hooks).
6. **`src/queries/listSdks.ts`** — `listSdks(db): SdkSummary[]`.
7. **`src/queries/queryTypes.ts`** — `queryTypes(db, { pattern, sdk?, kind? }): SymbolMatch[]` using FTS5 `MATCH` syntax.
8. **CLI surfacing**: `code-mode list-sdks` and `code-mode query-types <pattern>` — useful for manual testing before MCP wiring.

### Success Criteria:

#### Automated Verification:
- [ ] DB tests pass: `bun test test/db`
- [ ] Migration runner is idempotent: running `bun test test/db/migrate.test.ts` twice against the same DB leaves `PRAGMA user_version` unchanged
- [ ] Adding a stub `002_noop.sql` bumps the version pragma
- [ ] Indexer tests pass on a fixture workspace: `bun test test/index`
- [ ] Round-trip: `code-mode init && code-mode reindex && sqlite3 .code-mode/code-mode.db "select count(*) from symbols"` > 0
- [ ] `code-mode list-sdks` includes `stdlib`
- [ ] `code-mode query-types filter` returns the `filter` signature
- [ ] Deleting a stdlib file + re-running reindex removes it from the DB

#### Manual Verification:
- [ ] Add a new SDK file by hand; reindex; `query-types` finds it.
- [ ] Symbol JSON returned by `query-types` includes JSDoc fields.

**Implementation Note**: Pause after Phase 4 — the DB schema is the hardest thing to change later. Confirm columns/tables feel right before building run/save/MCP on top.

## Phase 5 — `code-mode run` + `code-mode save`

### Goal
The type-safety gate. Both commands share a typecheck step; `run` executes via Bun subprocess and captures structured output; `save` validates then persists + re-indexes.

### Changes

1. **`src/runner/loader.ts`** — template emitted inline at runtime (no file on disk required). Imports the target script, calls its default export (`async function main(args)`), serializes the result, prints `\n__CODE_MODE_RESULT__:<json>\n` on stdout. Any other stdout/stderr are captured as logs.
2. **`src/runner/limits.ts`** — execution controls (configurable at call-site and via workspace `.code-mode/config.json`):
   - **Timeout**: wall-clock, `AbortSignal.timeout()` wired to `Bun.spawn({ signal })`. Default 30s, max 300s, override via `--timeout <ms>` or config.
   - **Memory**: `ulimit -v <kb>` applied in a shell wrapper that precedes the Bun invocation on POSIX. Default 512MB, max 2GB, override via `--max-memory <mb>`. On Windows: documented as unsupported (MVP), warning logged.
   - **CPU seconds**: `ulimit -t <seconds>` in the same wrapper. Default 60, override via `--max-cpu <s>`.
   - **Output caps**: hard-cap captured stdout+stderr at 1MB (default) / `--max-output <bytes>`. Exceeding truncates with a `[truncated]` marker in `logs`.
   - **Args size**: reject `argsJson` > 256KB before spawning (prevents memory blowout from pathological inputs).
   - All limits recorded in the `RunResult` so callers can tell a timeout apart from a memory kill.
3. **`src/runner/exec.ts`** — `execScript({ workspaceDir, entry, argsJson, limits? }): Promise<RunResult>`:
   - Typecheck `entry` + its transitive imports (ts-morph `getReferencedSourceFiles` walk, depth-limited).
   - If diagnostics: return `{ success: false, diagnostics }` without spawning.
   - Else: compose POSIX wrapper (`sh -c 'ulimit -v N -t M; exec bun run -- <loader> <entry> <argsJson>'`) and `Bun.spawn(...)` with captured stdio + `AbortSignal.timeout()`.
   - Parse sentinel line → `result`; rest → `logs`. Distinguish `{ reason: 'timeout' | 'memory' | 'cpu' | 'crash' | 'ok' }` based on signal / exit code.
4. **`src/commands/run.ts`** — CLI entry supporting:
   - `code-mode run --inline <file.ts> [--args '{}']` — runs a file outside `.code-mode/` (ad hoc).
   - `code-mode run --source - [--args '{}']` — reads source from stdin, writes to a tempfile.
   - `code-mode run <name> [--args '{}']` — resolves `.code-mode/scripts/<name>.ts`.
   - Flags: `--timeout`, `--max-memory`, `--max-cpu`, `--max-output`. Per-run overrides beat workspace config.
   - On success, increments `runs`, updates `last_run`, rolling-averages `success_rate` in the DB.
5. **`src/commands/save.ts`** — `code-mode save <name> --file <path>` or `--source -`:
   - Writes to `.code-mode/scripts/<name>.ts` (rejects if exists unless `--overwrite`).
   - Runs typecheck; if diagnostics, **removes the just-written file** and returns them.
   - On success, calls `reindex({ paths: [savedPath] })`.
6. **Script contract documented** in `README.md` or a new `docs/scripts.md`: `export default async function main(args: unknown): Promise<unknown>`.

### Success Criteria:

#### Automated Verification:
- [ ] Runner tests pass: `bun test test/runner`
- [ ] Golden: save a stdlib-using script, run it with args, assert structured result
- [ ] Broken save is rejected and the file is not left behind
- [ ] Broken inline run returns `success: false` with diagnostics, does not spawn
- [ ] Usage counters increment exactly once per successful run
- [ ] Timeout enforced: script with `while(true){}` returns `{ success: false, reason: 'timeout' }` within `--timeout + 1s`
- [ ] Memory cap enforced (POSIX): script allocating past `--max-memory` exits with `reason: 'memory'`
- [ ] Output cap enforced: script writing >1MB to stdout has `logs` truncated with `[truncated]` marker
- [ ] Oversize args rejected pre-spawn: `argsJson` > 256KB returns error without spawning

#### Manual Verification:
- [ ] Write a real filter script that imports `@/sdks/stdlib/filter`, save it, run it.
- [ ] Compose: save a second script that imports the first; confirm the transitive typecheck catches a mismatch.
- [ ] Timeout kicks in on an infinite loop.
- [ ] Pass a hostile `argsJson` (deeply nested object, large string) — confirm limit rejects it cleanly.

**Implementation Note**: Pause after Phase 5 — this is the core value prop. Exercise it manually against at least one real-world script shape before moving on to codegen.

## Phase 6 — MCP SDK generator

### Goal
Detect each connected MCP server, introspect its `tools/list`, and emit a typed TS SDK module per server into `sdks/.generated/` that scripts can import and call.

### Changes

1. **`src/sdk-gen/config.ts`** — discover MCP servers in priority order:
   - Explicit: `--mcp-config <path>` on reindex.
   - Project: `./.mcp.json`.
   - User: `~/.claude.json` mcpServers.
   - Returns `McpServerSpec[]` with `{ name, transport: 'stdio'|'http', command?, args?, env?, url? }`.
2. **`src/sdk-gen/introspect.ts`** — for each spec:
   - Spin up an MCP client against the server (`@modelcontextprotocol/sdk`).
   - Send `initialize` + `tools/list`.
   - Collect `{ name, description, inputSchema, outputSchema? }`.
   - Handle timeouts + process cleanup. Failures don't abort the whole reindex — they're recorded and surfaced in `doctor`.
3. **`src/sdk-gen/codegen.ts`** — JSON Schema → TS string:
   - Primitives map straight through; `object` → `interface`; `array` → `T[]`; `enum` → union of string literals; `any`/missing → `unknown`.
   - Nested `$ref` resolved inline (MVP). Depth cap to 4 to avoid pathological schemas.
   - Emits one named interface per tool input + per tool output, and a callable wrapper.
4. **`src/sdk-gen/emit.ts`** — writes:
   - `sdks/.generated/_client.ts` — shared runtime helper that spawns the relevant MCP server on demand, runs one tool call, returns the parsed result. Cached per-process.
   - `sdks/.generated/<server>.ts` — one file per server, importing from `_client.ts`. Example shape:
     ```typescript
     /** @generated from MCP server "github" at 2026-04-13T... */
     import { callTool } from "./_client";
     export interface CreateIssueArgs { /* ... */ }
     export interface CreateIssueResult { /* ... */ }
     /** Create a GitHub issue */
     export async function createIssue(args: CreateIssueArgs): Promise<CreateIssueResult> {
       return callTool("github", "create_issue", args) as Promise<CreateIssueResult>;
     }
     ```
   - Entire `.generated/` dir is wiped before rewriting — it's derived state.
5. **Hook into reindex** — `src/commands/reindex.ts` calls `sdk-gen` first (unless `--no-sdk-gen`), then the filesystem walker picks up the generated files as any other SDK.

## Phase 7 — `code-mode mcp` — MCP server mode

### Goal
Expose the five MCP tools (`search`, `run`, `save`, `list_sdks`, `query_types`) over stdio, delegating to Phase 4–6 internals. No new behavior — just an adapter layer.

### Changes

1. **`src/queries/search.ts`** — `search(db, { query, scope?, kind? }): SearchHit[]`. Queries both `scripts_fts` and `symbols_fts`, merges with a simple score (FTS rank + usage boost), filters `status='ok'` scripts. Returns pointers (path, name, description, scope, score) — no source.
2. **`src/mcp/server.ts`** — sets up `Server` from `@modelcontextprotocol/sdk/server`. Registers five tools with JSON Schema for each arg:
   - `search({ query: string, scope?: 'script'|'sdk'|'stdlib'|'generated', kind?: string }) → { results: SearchHit[] }`
   - `run({ mode: 'named'|'inline'|'stdin', name?, source?, argsJson? }) → RunResult`
   - `save({ name, source, description?, tags?, overwrite? }) → { path, diagnostics? }`
   - `list_sdks({}) → { sdks: SdkSummary[] }`
   - `query_types({ pattern, sdk?, kind? }) → { matches: SymbolMatch[] }`
3. **`src/mcp/handlers/*`** — one file per tool. Each is a thin function that parses args, calls the Phase 4–6 internals, returns the result.
4. **`src/commands/mcp.ts`** — sets up stdio transport, constructs the workspace dir (defaults to `cwd`), starts the server.
5. **Protocol polish**:
   - Every tool returns MCP-shaped content (JSON stringified in a `text` block for broad compat) — the brainstorm Q "How do scripts produce output?" already mandates structured + logs, so we return both channels.
   - Errors surface as `isError: true` with the diagnostic message.

#### Automated Verification:
- [ ] MCP tests pass: `bun test test/mcp`
- [ ] Integration: spawn `bun run bin/code-mode.ts mcp` as a subprocess, connect with the MCP client SDK, call each of the 5 tools, assert non-error responses

## Phase 8 — `doctor` + `gc` + broken-script hook + MCP starter templates

### Goal
Polish. Health + cleanup + self-healing via a documented Claude Code hook. Starter templates per connected MCP help agents bootstrap usage.

### Changes

1. **`src/commands/doctor.ts`** — runs a full typecheck pass:
   - Every script: on diagnostics, set `status='unusable'`, `status_reason=<first message>` in DB.
   - Reports (stdout + JSON via `--json`):
     - Broken scripts (count + paths)
     - Stale scripts (`last_run` older than `--stale-days` default 30)
     - Low success scripts (`success_rate < 0.5`, `runs >= 3`)
     - DB freshness mismatch (files newer than `indexed_at`)
   - Exit code 1 if broken scripts exist (CI-friendly); `--no-fail` to override.
2. **`src/commands/gc.ts`** — soft cleanup:
   - Duplicate detection: group symbols by normalized signature (strip whitespace, reorder unions alphabetically), list candidates for consolidation.
   - Stale detection: scripts meeting both "never run OR last_run > N days" AND "no incoming imports".
   - Default dry-run; `--apply` moves flagged files to `.code-mode/.trash/<timestamp>/` (not `rm`). Trash purging is manual.
3. **Broken-script hook — documented, not baked in**:
   - Ship a `hooks/post-edit.json` example snippet for Claude Code settings that runs `code-mode reindex --paths <changed>` on Write/Edit under `.code-mode/**`.
   - README section "Integrate with Claude Code" with the snippet + install instructions.
   - `reindex` with `--paths` already handles the incremental typecheck in Phase 4, so no new runtime work.
4. **Search exclusion**: update Phase 7 `search` to filter `status='ok'`. (Technically could have been done in Phase 7; doing it here keeps the status column story self-contained.)
5. **MCP starter templates** — in `src/sdk-gen/emit.ts` (called from reindex):
   - After emitting `.generated/<server>.ts`, check if `.code-mode/scripts/<server>/` is empty.
   - If empty and `--templates` (default on), emit 1 starter script per server: `scripts/<server>/example.ts` that imports from the generated SDK and calls the first tool with a placeholder argument. These count as real scripts and get indexed.
   - `--no-templates` opts out.

## Phase 9 — Web inspector

### Goal
A browser-based inspector (à la `@mcp-use/inspector` — "Swagger UI for MCP") served by `code-mode inspect`. Lets developers browse every configured MCP's tools/resources/prompts, inspect schemas, invoke tools live, and watch `.code-mode/sdks/.generated/` being produced. Also doubles as a debugger for our own MCP server.

### Scope clarification
- **In**: tools (list + invoke), resources (list + fetch), prompts (list), connection health, generated SDK preview per-server.
- **Out**: authoring UI for scripts (use CLI/editor), time-travel debugging, persisted history beyond current session, multi-user.

### Changes

1. **Packaging** — inspector lives in its own directory `packages/inspector/` (monorepo pivot — root `package.json` grows a `workspaces: ["packages/*"]` entry; core CLI moves to `packages/core/`). This keeps the React bundle out of the core install path when the user doesn't need it.
2. **Inspector UI** (`packages/inspector/ui/`) — Vite + React:
   - Sidebar: list of configured MCP servers + local `code-mode mcp`.
   - Main pane (per server): tabs for Tools / Resources / Prompts / Generated SDK.
   - Tools tab: schema panel + form-driven invocation (schema → form via a small JSON-Schema→form helper; reject anything non-trivial to a raw JSON textarea fallback). Result + logs below.
   - SDK tab: read the emitted `.code-mode/sdks/.generated/<server>.ts` and display with syntax highlighting — lets devs eyeball codegen quality.
   - No server-side state; everything is per-session.
3. **Inspector server** (`packages/inspector/server/`) — tiny Bun HTTP server:
   - Serves the built React bundle.
   - `GET /api/servers` — list configured MCP servers.
   - `POST /api/invoke` — `{ server, tool, args }`, proxies to the MCP server via `@modelcontextprotocol/sdk` client, streams result back.
   - `GET /api/generated/:server` — returns file contents of `.generated/<server>.ts`.
   - No auth (localhost-only binding by default: `127.0.0.1`). `--host 0.0.0.0` opts into LAN exposure with a warning.
4. **`src/commands/inspect.ts`** (in core) — `code-mode inspect [--port 3456] [--host 127.0.0.1] [--no-open]`:
   - Spawns the inspector server as a subprocess from the `@code-mode/inspector` package (`bunx -p @code-mode/inspector code-mode-inspect …`) — lazy load, not bundled into the core binary.
   - Opens the browser unless `--no-open`.
   - Ctrl-C tears down server + any child MCP clients.
5. **Reuse from earlier phases**:
   - Introspection from Phase 6 (`src/sdk-gen/introspect.ts`) — the inspector's `/api/servers` + tool-invocation machinery call into this module. No reimplementation.
   - Generated SDK files from Phase 6 surface verbatim in the SDK tab.
6. **Stretch (only if Phase 9 completes quickly)**:
   - Watch mode: re-run `code-mode reindex` when `.mcp.json` changes, live-refresh the sidebar.
   - Hit-history: in-memory log of prior tool invocations during the session.

### Success Criteria:

#### Automated Verification:
- [ ] Inspector suites pass: `bun test test/inspector`
- [ ] Lazy-load check: `bunx code-mode --help` runtime doesn't load React (grep the module graph or spot-check install time without the inspector workspace)
- [ ] UI unit tests (component-level for the invocation form + result panel): `cd packages/inspector/ui && bun test`
- [ ] `code-mode inspect --port 0 --no-open & sleep 2 && curl localhost:$PORT/api/servers | jq .` returns a JSON array
- [ ] Tool invocation round-trip against the fake MCP fixture succeeds: `POST /api/invoke` → expected result shape

## Manual E2E Walkthrough

Two complementary walk-throughs — first a human driving the CLI, then a programmatic MCP client driving `code-mode mcp`. Both operate inside throwaway tmp folders so your machine stays clean. Substitute `code-mode` with `bunx -p /path/to/this-repo code-mode` if the CLI isn't on your PATH yet.

### E2E A — CLI (user perspective)

```bash
# Scratch workspace
E2E_DIR="$(mktemp -d -t code-mode-e2e-cli.XXXX)"
cd "$E2E_DIR"

# 1. Fresh init
code-mode init
test -f .code-mode/code-mode.db

# 2. Reindex picks up stdlib
code-mode reindex
code-mode list-sdks                  # expect: stdlib
code-mode query-types filter         # expect: stdlib/filter signature

# 3. Save + run a typechecking script
cat > /tmp/sample.ts <<'EOF'
import { filter } from "@/sdks/stdlib/filter";
export default async function main(args: { xs: number[] }) {
  return filter(args.xs, (n) => n > 2);
}
EOF
code-mode save sample --file /tmp/sample.ts
code-mode run sample --args '{"xs":[1,2,3,4]}'       # expect: result: [3,4], success: true

# 4. Save rejects a broken script
cat > /tmp/broken.ts <<'EOF'
export default async function main(args: string) {
  return args.toUpperCase().nope();
}
EOF
code-mode save broken --file /tmp/broken.ts           # expect: nonzero exit, diagnostics, no file

# 5. Execution limits enforced
cat > /tmp/loop.ts <<'EOF'
export default async function main() { while (true) {} }
EOF
code-mode save loop --file /tmp/loop.ts
code-mode run loop --timeout 2000                     # expect: success:false, reason:'timeout'

# 6. Reindex generates SDKs from configured MCPs
# (pre-req: the scratch dir has a .mcp.json or ~/.claude.json is set up)
code-mode reindex
ls .code-mode/sdks/.generated/                        # expect: <server>.ts + _client.ts
code-mode list-sdks                                   # expect: stdlib + each MCP server
code-mode query-types <known-tool-name>

# 7. Inspector (Phase 9)
code-mode inspect --port 3456 &
INSPECTOR_PID=$!
open http://localhost:3456                            # browse tools, invoke one interactively
kill $INSPECTOR_PID

# 8. Doctor + gc
code-mode doctor                                      # expect: exit 0
sed -i '' 's/n > 2/n > "2"/' .code-mode/scripts/sample.ts
code-mode reindex --paths .code-mode/scripts/sample.ts
code-mode doctor                                      # expect: exit 1, sample flagged unusable
code-mode gc --stale-days 0                           # expect: dry-run list

# Cleanup
cd / && rm -rf "$E2E_DIR"
```

### E2E B — MCP client (programmatic)

```bash
# Scratch workspace
E2E_DIR="$(mktemp -d -t code-mode-e2e-mcp.XXXX)"
cd "$E2E_DIR"
code-mode init
code-mode reindex
```

Then, from a driver script (can live in `test/e2e/mcp-client.test.ts` and be invoked via `bun test test/e2e/mcp-client.test.ts --workspace "$E2E_DIR"`):

```ts
import { Client } from "@modelcontextprotocol/sdk/client";
import { StdioClientTransport } from "@modelcontextprotocol/sdk/client/stdio";

const transport = new StdioClientTransport({
  command: "bunx",
  args: ["-p", CODE_MODE_REPO, "code-mode", "mcp"],
  env: { ...process.env, CODE_MODE_WORKSPACE: process.env.WORKSPACE! },
});
const client = new Client({ name: "e2e", version: "0" });
await client.connect(transport);

// 1. tools/list — expect 5 tools
const { tools } = await client.listTools();
assertEqual(tools.map(t => t.name).sort(),
  ["list_sdks","query_types","run","save","search"]);

// 2. list_sdks — stdlib present
const sdks = await client.callTool({ name: "list_sdks", arguments: {} });
assert(sdks.content[0].text.includes("stdlib"));

// 3. search — find stdlib filter
const hits = await client.callTool({
  name: "search",
  arguments: { query: "filter array" },
});
assert(hits.content[0].text.includes("stdlib/filter"));

// 4. save — rejects broken, accepts valid
const bad = await client.callTool({
  name: "save",
  arguments: { name: "bad", source: "export default async function main(){return x}" },
});
assert(bad.isError === true);

const good = await client.callTool({
  name: "save",
  arguments: {
    name: "double",
    source: `export default async function main(a:{n:number}){return a.n*2}`,
  },
});
assert(!good.isError);

// 5. run — named, inline, and resource-constrained
const runResult = await client.callTool({
  name: "run",
  arguments: { mode: "named", name: "double", argsJson: '{"n":21}' },
});
assert(JSON.parse(runResult.content[0].text).result === 42);

// 6. query_types — generated SDK lookup
const types = await client.callTool({
  name: "query_types",
  arguments: { pattern: "filter" },
});
assert(JSON.parse(types.content[0].text).matches.length > 0);

await client.close();
```

Post-run cleanup: `rm -rf "$E2E_DIR"`.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| ts-morph cold-start too slow on even small workspaces | Fall back to `@ts-morph/bootstrap` (Phase 3 checkpoint) |
| `bun install` inside `.code-mode/` is slow or noisy on `init` | `--no-install` flag; only needed when stdlib pulls external deps (which it doesn't in MVP) |
| FTS5 ranking is mediocre — agent can't find scripts by intent | Usage-based re-ranking at query time; semantic search deferred to post-MVP |
| MCP server subprocess leaks on introspection failure | `finally`-block process cleanup; integration test with a deliberately-hanging server |
| Generated SDKs break when upstream schema changes | `.generated/` is wiped each reindex; regeneration is the contract |
| Bun TypeScript CJS interop on `typescript` | Confirmed via research: always `import * as ts from "typescript"` |
| Schema evolution without an ORM gets painful | Versioned `.sql` files + `PRAGMA user_version` runner from Phase 4. Swap in Drizzle later if churn actually hurts. |
| `ulimit` unavailable on Windows — no memory/cpu caps | Documented as MVP limitation. Timeout + output cap still enforced everywhere. |
| Inspector React bundle balloons binary size | Ship as separate `@code-mode/inspector` package loaded lazily on `code-mode inspect`; not bundled into the core CLI. |

## Open Tracks (Out of Scope)

These are intentionally deferred; listed so we don't lose them:

- Global scope (`~/.code-mode/`) with auto-git.
- Sandboxed execution (e2b, modal, just-bash).
- Semantic search.

- Skills for harnesses.
